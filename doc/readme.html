1. Compiling

To compile, open the src directory in the terminal, and type make.

2. Command-line usage

The following switches apply to all modes:

--help
Show basic usage information.
--version
Print version and exit.
--cli
Do not launch GUI interface.
--gui
Use GUI rather than CLI version (this is the default).
--rotorpos R
Position to start the rotor from. Default is A
--output file
File to dump output to. Use "-" for standard output.
--verbose
Print extra information which may be useful for debugging unexpected output.
--html
Used in conjunction with --verbose to get tables in HTML rather than plaintext.

2.1 Simulator mode

--simulate file.machine
File to read machine definiton from. Set to "-" to generate a random machine definition.
--encipher | --decipher | --new
Use encipher or decipher mode, or create a new machine and output it. Implies --cli.
--input file
Input file for the operation, or "-" to read from standard input (this is the default). Only used for --encipher or --decipher.

2.2 Ciphertext/plaintext decoder mode

--known-plaintext
Attempt to figure out wiring based on plaintext/ciphertext. You will also be prompted for a rotor position if --rotorpos is not specified. The output of this operation is a machine file suitable for the simulator.

3. Machine files

In order to exchange enciphered messages, both the sender and receiver must have:

  1. Machines with the same wiring
  2. The same rotor starting-position

This program will load and save files in .machine format, which is simply a startup script. Use one of the example machines from the files folder to get started.

You should generate your own random .machine file to encipher messages with:

java RotorCipher --new --output myfile.machine

4. Usage examples

4.1 Enciphering and deciphering messages (GUI)

To launch the GUI using one of the example machines, use this command from the bin folder:

java RotorCipher --simulate ../files/week01.machine

4.2 Interactive command-line mode

To launch the same simulator in command-line mode, tack --cli option onto the command:

java RotorCipher --cli --simulate ../files/week01.machine

We can then encipher "testing testing" at Rotor position B: 

Encipher, decipher, or quit? (e / d / q): e
Set rotor: B
Plaintext: testing testing
CJEULIJ HHYORFP

And as a sanity check, we can try to return to the plaintext from this output:

Encipher, decipher, or quit? (e / d / q): d
Set rotor: B
Ciphertext: CJEULIJ HHYORFP
testing testing

It is worth noting that without the correct rotor position, random junk will be given:

Encipher, decipher, or quit? (e / d / q): d
Set rotor: C
Ciphertext: CJEULIJ HHYORFP
xhyzdub ezggmak

4.3 Non-interactive mode

You can also read from a file, enciphering or deciphering it line-by-line. After each line, the rotor is returned to the starting position.

For example, to encipher "Hello this is a message" and store it to foo.txt, we could use this rather long invocation:

echo "Hello this is a message" | java RotorCipher --simulate ../files/week02a.machine --rotorpos M --encipher --output foo.txt

And sure enough, our enciphered message is now securely stored in foo.txt:

$ cat foo.txt 
QCVXA MUGY FQ P TDHUTFF

To decode the message, we use --input foo.txt, and change the --encipher switch to --decipher: 

$ java RotorCipher --simulate ../files/week02a.machine --rotorpos M --decipher --input foo.txt
hello this is a message

4.4 Using with minimodem to transmit messages

If you want to encipher some communications over-the-wire, then pipe output to minimodem like this: 

java RotorCipher --simulate ../files/week02a.machine --rotorpos q --encipher | minimodem --tx 30

And on the other end, use --decipher with minimodem's --rx switch:

minimodem --rx 30 -q | java RotorCipher --simulate ../files/week02a.machine --rotorpos q --decipher

4.5 Finding rotor wiring from plaintext/ciphertext pair

If you have plaintext and corresponding ciphertext, then you can find out how the machine was wired. This deature is only implemented in CLI mode. You can pass it the example enciphering to see how it works:

java RotorCipher --known-plaintext --cli --output week02a-deciphered.machine < ../files/week02a-decipher-input.txt

5. Algorithms

5.1 Enciphering and deciphering messages

Messages will be enciphered and deciphered according to the wiring of an individual machine. The rotor is mobile and has 26 states.

A rotor's wiring is represented as a series of pairs, as they stand in the A position.

When enciphering at position A, the pair Qf means that when enciphering f → Q, and when deciphering Q → f.

Each pair that is specified will be loaded into two tables. One for enciphering, and the other for deciphering. Qf looks like this:

Enciphering table for a rotor with pair Qf at positon A
abcdefghijklmnopqrstuvwxyz
A Q
Deciphering table for a rotor with pair Qf at positon A
ABCDEFGHIJKLMNOPQRSTUVWXYZ
A f

As the rotors move, the effect of this wire on positions B-Z is predictable. With each rotation, the letter affected changes, and the plaintext/ciphertext character returned goes down the alphabet one character

We need 26 defined pairs to fill a cipher table. When no pairs are defined, the default definition maps every letter to itself and will not transform the text at all.

In addition to this, input and output wiring can be added. This is functionally the same as an extra rotor on either side, but only position A is ever used (ie, it does not rotate).

The full process for enciphering a single character is then:

  1. Move the rotor one position
  2. Encipher the character through the input wiring
  3. Encipher the character through the rotor
  4. Encipher the character through the output wiring

And similarly to decipher:

  1. Move the rotor one position
  2. Decipher the character through the output wiring
  3. Decipher the character through the rotor
  4. Decipher the character through the input wiring

5.2.1 Building a machine file from the cipher table

The above is quite sufficient for hand-encoding and decoding of messages. This next section deals with the simple task of writing the table to a .machine file. There are a few more things which we need to calculate at this point:

  1. Which letters cannot be enciphered by this input wiring. These are the ones which do not appear in the top row: z v j x q k
  2. Which characters cannot be deciphered by this input wiring. These letters appear in the second-top row with a blank above them: D R U V X Y

We now disregard most of the table, and use only the top rows:

 edg rsbfuwlchmtip ny a o
 ABCDEFGHIJKLMNOPQRSTUVWXYZ
AZEJNTIROYHCMWXKGVLDSPFQABU

Reading across the top two rows and skipping blanks, read each pair of letters like this: 

A from e (AE), B from d (BD), C from g (CG), E from r (ER) ... Z from o (ZO)

Put these together in pairs to form the following input wiring string:

AE BD CG ER FS GB HF IU JW KL LC MH NM OT PI QP SN TY WA ZO

Now read the blanks and put an underscore after them instead — No inputs lead to these letters:

D_ R_ U_ V_ X_ Y_ (D from nothing, R from nothing, etc)

And those letters which don't appear in the top row are instead prefixed with an underscore — These letters return nothing:

_Z _V _J _X _Q _K (Nothing from Z, nothing from V, nothing from J, etc)

This is now the inWiring line of the machine file:

# Input wiring
inWiring "AE BD CG ER FS GB HF IU JW KL LC MH NM OT PI QP SN TY WA ZO D_ R_ U_ V_ X_ Y_ _Z _V _J _X _Q _K"

Using the same process, we read the second and third lines together. If there are blanks on this line, use underscores the same as above (but your machine may not work very well). In our case we have a full wiring, so we come up with this line:

# Rotor wiring
rotor 1 name "Rotor 1"
rotor 1 wiring "ZA EB JC ND TE IF RG OH YI HJ CK ML WM XN KO GP VQ LR DS ST PU FV QW AX BY UZ"

Machines must also be named, and given a default starting position:

# Machine name
name "Machine 1"

# Default config
selectRotors 1
setRotorsTo F

The following complete .machine file can now be run in the simulator to decipher any more messages we pick up:

# Machine name
name "Machine 1"

# Input wiring
inWiring "AE BD CG ER FS GB HF IU JW KL LC MH NM OT PI QP SN TY WA ZO D_ R_ U_ V_ X_ Y_ _Z _V _J _X _Q _K"

# Rotor wiring
rotor 1 name "Rotor 1"
rotor 1 wiring "ZA EB JC ND TE IF RG OH YI HJ CK ML WM XN KO GP VQ LR DS ST PU FV QW AX BY UZ"

# Default config
selectRotors 1
setRotorsTo F

6. Limitations and practical considerations

It should go without saying that this program is not intended to facilitate secure communications — It is a demonstration of one extremely dated type of polyalphabetic substitution cipher, and should not be used with any expectation of security.

This program will re-use the same configuration many times. For example, when piping output to another program, it will re-set the rotor to the starting position after each line. This is done to make it easy for programs to recover from errors by simply starting a new line, but the predictability of the enciphering creates a major weakness of its own.

The known-plaintext algorithm relies on unmixed output, by using the alphabet as a pattern. The simulator will happily mix outputs if outWiring is set in the machine file, but the --new option will always generate a machine with unmixed output, so that the known-plaintext algorithm can be used.

The known-plaintext algorithm processes columns individually. It could be improved so that after the table is filled, some of the columns with blank headers (those with only one or a few data points) are placed using this reduced search space.

In general, short messages containing pseudo-grable, enciphered with random machines (with the addition of mixed output) would have the lowest chance of being deciphered.
